How I Built It — Technical

The Grant Fit Scanner

Written for someone who already knows how to use Claude. This documents the actual prompting and workflow, step by step, so it can be reproduced or adapted.

The setup

I gave Claude a single upfront prompt that specified the full pipeline and the order of operations, rather than prompting turn-by-turn. This matters: for a multi-stage task (gather profile → research → build tool → explain → suggest next steps), front-loading the sequence in one message keeps Claude from skipping steps or collapsing them together.

The prompt specified:

  1. Ask for the nonprofit profile first (or offer a sample if I didn't have a real org)
  2. Actually search the web for real grants — no invented ones
  3. Build the results as an interactive artifact, not a text list
  4. Explain the design choices in plain English
  5. Suggest realistic "next feature" ideas

I also explicitly flagged the thing most likely to fail silently: "if it doesn't search on its own, just say 'search the web for real grant opportunities' again." That's a fallback instruction for a known failure mode — Claude can sometimes answer research-shaped questions from training knowledge instead of using the search tool, and calling that out upfront reduces the odds of it happening.

Step 1 — Profile elicitation

Claude asked whether I had a real org or wanted a sample profile. I chose sample. It generated a plausible mid-sized Ohio food bank (mission statement, four program areas, budget, location) and used that as the fixed research target for the rest of the session. Key point: I let it define the sample rather than specifying one myself, since the goal was speed, and a generic "food bank" profile is realistic enough to research against meaningfully.

Step 2 — Actual web research, not synthesis

This was the step to watch closely. I asked Claude to run real web_search calls against the profile — not to write plausible-sounding grant names from memory. It ran roughly six distinct searches, each narrower than the last:

  • general hunger-relief grants
  • a named corporate funder (Kroger Zero Hunger | Zero Waste) for eligibility and deadline specifics
  • a second named corporate funder (Walmart Foundation)
  • state-specific grants (Ohio)

Each search pulled from primary or near-primary sources (funder websites, Instrumentl, GrantWatch, Candid/Foundation Directory) rather than aggregator blog posts alone. I could verify this because Claude cited its sources inline with the specific claims, which is the tell that it's actually grounding the output rather than hallucinating funder names.

The output included two intentionally low-fit grants (a hyper-local Ohio Valley community foundation fund, and a USDA retail-expansion program) that read as close matches on paper but fail on geography or program structure. Including near-misses, not just wins, is what makes a fit-scoring system credible — if I'd only asked for "grants that match," Claude would have had no incentive to surface the ones that don't, and the tool would look like a keyword search wearing a scoring badge.

Step 3 — Artifact generation

I asked for "a simple interactive artifact (a webpage/tool I can actually use and click through)" rather than a document. Claude built a single self-contained HTML file with embedded CSS and JS — no external dependencies beyond a Google Fonts import — containing:

  • a static JS array of the 8 real grants (name, funder, deadline, amount, fit tier, numeric score, rationale, caveat)
  • filter buttons (All / High / Medium / Low)
  • a sort toggle (deadline soonest/furthest)
  • a live search box
  • render logic that re-filters and re-sorts on every interaction

Because this was a one-off deliverable rather than a component meant for reuse in a larger app, plain HTML/CSS/JS was the right call over React — no build step, opens directly in a browser, and is trivial to hand off or extend later.

Step 4 — The explanation layer

I asked Claude to explain, in plain English, why it organized the tool the way it did — deadline-first sorting, the four-factor fit score, the flag system — specifically so I could describe the reasoning intelligently in an application essay without having originated the design logic myself. This is worth doing deliberately with Claude: ask it to externalize its own design rationale as a separate deliverable, not just build silently.

Step 5 — Forward-looking features

Finally I asked for 2–3 realistic "next features" — deadline auto-tracking, LOI drafting, an outcome-tracking feedback loop — framed as things worth mentioning in an interview to demonstrate roadmap thinking, without requiring me to actually build them in this pass.

Reusable pattern

The general shape here — profile/context gathering → live research with real tool calls → interactive artifact → self-explanation → forward-looking roadmap — is a reusable prompt structure for any "research this, then build a tool around what you found" task. The two levers that mattered most were (1) explicitly demanding real search over synthesized-sounding output, and (2) asking for the design rationale as a distinct, separate output rather than assuming I'd be able to reverse-engineer it from the artifact alone.
